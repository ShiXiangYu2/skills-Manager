## 网文逻辑审查修复（组合调用）

### 触发关键词
帮我检查下小说有没有bug、看看时间线有没有矛盾、人物有没有OOC、找剧情前后冲突、梳理伏笔有没有回收、检查小说剧情合理性、底线问题检查、小说剧情审查、大纲有问题要改、剧情矛盾怎么修、发现重复情节、设定崩坏了怎么办

### 核心功能

#### 组合调用设计

本技能是 `sumeru-audit` 和 `sumeru-fix` 的组合编排器，负责协调两个独立技能完成完整的审查修复流程：

**第一阶段：全局审查（调用 sumeru-audit）**
- 加载完整大纲和章节细纲，建立全局审查基准
- 分析整体剧情脉络和时间线结构
- 审查全局设定一致性（世界观、力量体系、规则设定）
- 识别主线支线关联问题和伏笔回收情况
- 检查整体冲突点分布和节奏把控
- **底线问题扫描**：全面扫描以下六类底线问题，生成 `bottom-line-checklist.json`：
  - 时间线矛盾（事件顺序错误、年龄/日期冲突）
  - 设定崩坏（力量体系前后不一致、世界观规则自相矛盾）
  - 人物OOC（性格突变无铺垫、行为与动机矛盾）
  - 重复情节（相似事件重复发生无差异、桥段雷同）
  - 信息泄露（角色知道不该知道的信息、信息边界混乱）
  - 伏笔死结（已埋伏笔无回收可能、伏笔自相矛盾）
- 记录**全局问题清单**到 `.sumeru/audit/global-issues.json`

**第二阶段：章节细节审查（调用 sumeru-audit，Agent Team 并行）**
- 使用多 Agent 并行处理，每个 Agent 负责一定数量的章节
- **⚠️ 遵循全局约束：每个子Agent最多负责3个章节**（详见 AGENTS.md "子Agent并行处理规则"）
- 所需Agent数 = ceil(总章节数 / 3)，分配策略为按章节顺序连续分配
- 为每章生成核心剧情概要（去掉细节，仅保留关键事件）
- 对每章进行详细审查：
  - 字数统计与填充需求识别
  - 时间线与事件时序验证
  - 人物行为与性格一致性（OOC检测）
  - 物品状态与信息边界检查
  - 场景描写与对话质量评估
  - 伏笔设置与回收状态
- 生成**章节概要**到 `.sumeru/audit/summaries/` 目录
- 记录**章节问题清单**到 `.sumeru/audit/chapter-issues/` 目录
- 支持断点续传，已审查章节可跳过

**第三阶段：统一修复执行（调用 sumeru-fix）**
- 合并全局问题和章节问题，按严重程度排序（致命 > 严重 > 中等 > 轻微）
- 制定并执行**修复计划**，分为三种修复类型：
  - **轻量修复**（直接执行）：文字修正、段落调整、语句优化、字数填充等 minor 修改，**直接修改 `chapters/` 文件**（修改前自动备份到 `.sumeru/write/original/`）
  - **严重问题闭环修复**（自动执行大纲修订+章节重写）：剧情逻辑严重矛盾、大面积OOC、设定崩坏、重复情节等需要重写的章节，**自动修订大纲并重写章节**
  - **底线问题专项修复**：六类底线问题必须全部解决，不允许跳过或搁置
- 执行轻量修复策略：
  - **全局问题修复**：调整整体时间线、统一设定、优化主线结构
  - **章节问题修复**：逐章修复字数、逻辑、OOC、剧情矛盾等轻量级问题
  - **联动修复**：处理跨章节的关联问题（如伏笔回收、人物成长）
- **执行严重问题闭环修复**：
  - **大纲修订**：直接修订 `.sumeru/outline/chapter-outlines.json` 中受影响章节的细纲，修正剧情矛盾、消除重复情节、修复设定冲突，修订内容记录到 `.sumeru/fix/outline-revisions.json`（包含修订原因、修订前后对比）
  - **章节重写**：基于修订后的细纲，使用子Agent并行重写对应章节（遵循3章/Agent约束），重写结果直接修改 `chapters/` 文件（修改前自动备份到 `.sumeru/write/original/`）
  - **修订一致性校验**：重写后的章节需与修订后的大纲进行一致性校验，确保大纲修订意图被正确落地
- **执行底线问题零遗漏核验**：
  - 逐一核对 `bottom-line-checklist.json` 中的每项底线问题
  - 轻量修复能解决的标记为「已解决-轻量修复」
  - 大纲修订+重写能解决的标记为「已解决-重写修复」
  - 无法自动解决的标记为「需人工干预」，并输出明确的**人工干预建议**（包含问题描述、影响范围、建议修改方向）
  - **全部底线问题标记为「已解决」或「需人工干预」后，审查流程才允许结束**
- 生成**修复报告**到 `.sumeru/fix/fix-report.json`
- 生成**修复计划** `fix-plan.json`，记录所有修复操作的详细信息
- 修复后重新审查，验证所有轻量修复和重写修复问题已彻底解决

#### 三阶段功能分配

**第一阶段：全局审查功能**
1. **整体剧情脉络分析**：审查主线、支线、伏笔的整体分布
2. **全局时间线校验**：梳理完整时间线，识别大跨度时序问题
3. **设定一致性检查**：验证世界观、力量体系、规则设定的统一性
4. **冲突点分布评估**：评估冲突强度、分布密度、节奏把控
5. **伏笔回收状态检查**：识别所有未回收的伏笔和回收方向

**第二阶段：章节细节审查功能**
6. **字数检查与填充**：确保每章字数达标，字数不足自动填充
7. **章节时间线与事件时序**：检查每章内部的时间逻辑和事件顺序
8. **OOC检测与人物一致性**：检查人物行为、性格、对话的一致性
9. **物品状态与信息边界**：追踪物品状态变化和人物信息边界
10. **场景描写与对话质量**：评估场景细节和对话的自然度
11. **章节伏笔设置**：记录每章的伏笔设置和预期回收位置
- **⚠️ 遵循全局约束：每个子Agent最多负责3个章节的审查**（详见 AGENTS.md）

**第三阶段：统一修复功能**
12. **问题统一修复**：合并所有问题，按严重程度排序，制定修复计划
13. **轻量修复执行**：文字修正、段落调整、语句优化、字数填充等直接修改 `chapters/` 文件
14. **大纲修订**：对严重有问题的章节，直接修订 `.sumeru/outline/chapter-outlines.json` 中对应细纲
15. **章节重写**：基于修订后的细纲，使用子Agent并行重写对应章节（遵循3章/Agent约束），直接修改 `chapters/` 文件
16. **底线问题零遗漏核验**：逐一核对六类底线问题，全部标记为「已解决」或「需人工干预」后才允许结束
17. **全局问题修复**：调整整体剧情、时间线、设定等大尺度问题
18. **章节问题修复**：逐章修复字数、逻辑、人物、场景等轻量级细节问题
19. **联动修复**：处理跨章节关联问题，确保修复后的整体一致性
20. **修复验证**：修复后重新审查，确保所有轻量修复和重写修复问题已彻底解决
- **⚠️ 轻量修复阶段遵循全局约束：每个Agent最多负责3个章节**（详见 AGENTS.md）

#### 修复机制说明
- **轻量级修复**：直接修改 `chapters/` 中有问题的段落或句子，修改前自动备份原始文件到 `.sumeru/write/original/`
- **严重问题闭环修复**：对严重有问题的章节（剧情矛盾、设定崩坏、重复情节等），自动修订 `.sumeru/outline/chapter-outlines.json` 中对应细纲，然后基于修订后的细纲使用子Agent并行重写章节，直接修改 `chapters/`（修改前自动备份到 `.sumeru/write/original/`），修订记录保存到 `.sumeru/fix/outline-revisions.json`
- **底线问题零遗漏**：六类底线问题（时间线矛盾、设定崩坏、人物OOC、重复情节、信息泄露、伏笔死结）必须全部解决或标记「需人工干预」，不允许跳过或搁置
- **自动备份**：修改 `chapters/` 文件前，自动将原始版本备份到 `.sumeru/write/original/`，确保可回滚
- **修复记录**：所有修复操作都记录在 `issues-fixed.json` 中，包含修复前后对比

### 输出内容
**第一阶段：全局审查输出**
- **全局剧情脉络分析报告**：主线、支线、伏笔的整体分布图
- **完整故事时间线图谱**：按时间轴整理的关键事件序列
- **设定一致性检查报告**：世界观、力量体系、规则设定的统一性分析
- **冲突点分布评估**：冲突强度、分布密度、节奏把控评估
- **全局伏笔回收状态表**：所有未回收伏笔清单与回收方向建议
- **底线问题清单**：六类底线问题的完整扫描结果（`bottom-line-checklist.json`）

**第二阶段：章节细节审查输出**
- **章节概要目录**：每章核心剧情概要（`.sumeru/audit/summaries/`）
- **字数检查报告**：每章字数统计、不达标的章节列表、填充改进记录
- **章节时间线与事件时序报告**：每章内部时间逻辑检查结果
- **OOC检测报告**：人物行为异常点、性格偏离分析
- **物品状态追踪表**：重要物品的获得、使用、丢失状态变化
- **章节伏笔设置记录表**：每章伏笔设置与预期回收位置
- **剧情连贯性评分**：多维度综合评分（时间线、逻辑性、人物一致性等）

**第三阶段：统一修复输出**
- **问题修复报告**：所有问题的修复情况记录、修复前后对比
- **大纲修订报告**：大纲修订内容、修订原因、修订前后对比（`outline-revisions.json`）
- **全局问题修复记录**：整体剧情、时间线、设定的调整记录
- **章节问题修复记录**：逐章修复的详细记录
- **联动修复说明**：跨章节关联问题的处理方案
- **底线问题核验报告**：六类底线问题的解决状态，「已解决」/「需人工干预」清单
- **修复验证报告**：修复后重新审查的验证结果
- **优化建议**：剧情调整方案、伏笔回收建议、冲突优化建议

### 数据持久化
**用户可见输出（当前工作目录）**：
- `剧情审查报告.md`：完整审查结果报告，包含所有问题、严重程度、修复建议（直接可读）
- `字数检查与填充报告.md`：字数统计、填充改进记录
- `全局审查报告.md`：第一阶段的全局分析结果
- `章节细节审查报告.md`：第二阶段的章节分析结果
- `统一修复报告.md`：第三阶段的修复记录，包含大纲修订、章节重写、底线问题核验结果

**中间数据（仅系统内部使用）**：
**第一阶段：全局审查数据（存于`.sumeru/audit/`目录）**
- `global-issues.json`：全局问题清单，包含时间线、设定、伏笔等问题
- `timeline.json`：完整时间线图谱，按时间轴排列的关键事件序列
- `plot-map.json`：剧情脉络图，显示主线、支线、冲突点分布
- `foreshadowing-tracking.json`：伏笔追踪表，包含所有伏笔的位置、内容、回收状态
- `coherence-score.json`：剧情连贯性评分明细
- `bottom-line-checklist.json`：**底线问题清单**，包含六类底线问题的扫描结果、状态追踪（待修复/已解决/需人工干预）

**第二阶段：章节细节审查数据（存于`.sumeru/audit/`目录）**
- `summaries/`：章节概要目录
  - `001.json`、`002.json`... 每章的核心剧情概要（章节号三位数零填充）
  - `summary-progress.json`：概要生成进度记录
- `chapter-issues/`：章节问题清单
  - `001.json`、`002.json`... 每章的详细问题记录（章节号三位数零填充）
- `word-count.json`：字数统计数据，每章字数、目标字数、填充情况

**第三阶段：统一修复数据（存于`.sumeru/fix/`目录）**
- `issues.json`：合并后的完整问题清单，按严重程度、类型分类
- `fix-report.json`：问题修复报告，包含修复前后对比
- `fix-plan.json`：修复计划，记录所有修复操作的详细信息
- `outline-revisions.json`：**大纲修订记录**，包含修订原因、修订前后对比、影响的章节列表
- `issues-fixed.json`：已修复问题记录
- `global-fix.log`：全局问题修复日志
- `chapter-fix.log`：章节问题修复日志
- `rewrite-chapters/`：**重写章节的修订记录**，包含重写前后对比

#### 与其他 Skill 配合
- **前置 Skill**：读取 `sumeru-outline` 的大纲数据和 `sumeru-write` 的章节数据
   - 使用 `.sumeru/outline/` 的世界观、人设、大纲作为基准
   - **使用 `.sumeru/outline/chapter-outlines.json` 作为预期剧情参考**
   - 使用 `chapters/` 目录下的章节内容进行审查
   - 将实际章节内容与细纲进行对比，识别剧情偏离

- **大纲修订能力**：review 发现严重问题时，直接修订 `.sumeru/outline/chapter-outlines.json` 中受影响章节的细纲
   - 修订内容记录到 `.sumeru/fix/outline-revisions.json`，包含修订原因、修订前后对比
   - 修订后的大纲供后续章节重写和后续流程使用

- **章节重写能力**：review 基于修订后的细纲，使用子Agent并行重写对应章节（遵循3章/Agent约束）
   - 重写结果直接修改 `chapters/` 文件（修改前自动备份到 `.sumeru/write/original/`）
   - 重写前后对比记录到 `.sumeru/fix/rewrite-chapters/`

- **修复计划输出**：修复完成后生成 `fix-plan.json`，记录所有修复操作的详细信息
   - worldbuilder 在编排流程时读取此文件，确认审查阶段的修复情况
   - 独立使用时，用户可查看 `fix-plan.json` 了解完整修复记录

- **后续 Skill**：
   - **sumeru-polish**：接收修复后的章节内容（已包含大纲修订+重写修复的结果），进行文笔润色
   - **sumeru-finalize**：接收修复后的章节内容，进行完稿校验和多平台导出

#### 三阶段数据流向
```
sumeru-write 章节
    ↓
第一阶段：全局审查（调用 sumeru-audit）
    ↓
第二阶段：章节细节审查（调用 sumeru-audit，Agent Team 并行）
    ↓
合并问题清单
    ↓
第三阶段：统一修复（调用 sumeru-fix）
    ├─ 轻量修复 → 直接修改 chapters/（自动备份到 .sumeru/write/original/）
    ├─ 严重问题 → 大纲修订（outline-revisions.json）→ 章节重写（自动备份）
    └─ 底线问题 → 零遗漏核验（bottom-line-checklist.json 全部「已解决」或「需人工干预」）
    ↓
供 polish 和 finalize 使用
```

#### 数据复用
- 返工修改时直接读取问题清单定位需要调整的章节
- 支持增量审查，新增章节时基于已有审查结果只检测新增内容
- 修复完成后可再次调用自动验证问题是否解决
- 字数填充记录可用于后续章节的字数参考
